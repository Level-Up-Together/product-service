## 3. 에러 포인트 (2개)

### 에러 1: 미션 검색 네이티브 쿼리 — `searchMissionsAdmin` (주요 원인)

**파일**: `MissionRepository.java:108-136`

```java
@Query(value = "SELECT * FROM mission m WHERE " +
    "(:keyword IS NULL OR LOWER(CAST(m.title AS TEXT)) LIKE ...) " +
    "AND (:source IS NULL OR m.source = CAST(:source AS TEXT)) " +
    // ... 나머지 조건들
    "ORDER BY m.created_at DESC",
    nativeQuery = true)
Page<Mission> searchMissionsAdmin(..., Pageable pageable);
```

**문제점 3가지**:

#### 3-1. `is_deleted` 필터 누락

네이티브 쿼리에 `AND m.is_deleted = false` 조건이 없다. 소프트 삭제된 미션이 결과에 포함되어:
- 삭제된 미션의 nullable 필드에서 **NPE 발생** 가능 (DTO 변환 시)
- 잘못된 데이터가 Admin에 노출

**비교**: `findAllByOrderByCreatedAtDesc(Pageable)` (검색 조건 없을 때 사용)는 JPA 메서드이므로 엔티티 전체를 반환하지만, 마찬가지로 `isDeleted` 필터가 없음

#### 3-2. 하드코딩된 `ORDER BY` + Pageable `Sort` 충돌

네이티브 쿼리에 `ORDER BY m.created_at DESC`가 하드코딩되어 있는데, Controller에서 `PageRequest.of(page, size, sort)`로 Pageable에도 Sort를 넣어 전달한다.

```java
// MissionAdminInternalController.java:48-50
Sort sort = "ASC".equalsIgnoreCase(sortDirection)
    ? Sort.by(sortBy).ascending()   // Sort.by("createdAt").descending()
    : Sort.by(sortBy).descending();
```

Spring Data JPA는 nativeQuery에 Pageable의 Sort를 **추가로 append**하려 시도하는데, 이때 `createdAt`이 실제 DB 컬럼명 `created_at`과 불일치하여 **SQL 오류** 발생 가능:

```sql
-- 생성되는 SQL (추정):
SELECT * FROM mission m WHERE ... ORDER BY m.created_at DESC, createdAt DESC
--                                                           ^^^^^^^^^ 존재하지 않는 컬럼
```

#### 3-3. countQuery에 ORDER BY 누락은 정상이지만 is_deleted 필터도 누락

countQuery (line 118-126)도 `is_deleted` 필터가 없어 삭제된 미션이 카운트에 포함됨 → 페이징 정보 불일치

### 에러 2: 카테고리 API 역직렬화 (부차적)

**호출**: `MvpMissionCategoryFeignClient.getAllCategories()` 또는 `getActiveCategories()`

**Feign 반환 타입**: `MvpApiResponse<List<MissionCategoryResponse>>`

Product의 `MissionCategoryInternalController`는 `ApiResult<List<MissionCategoryResponse>>` 형식으로 응답한다. Admin의 `MvpApiResponse<T>`가 이를 올바르게 역직렬화하지 못하면 에러 발생.

이 에러는 미션 관리 페이지 진입 시 카테고리 드롭다운 데이터를 가져오는 별도 API 호출에서 발생할 수 있으며, 미션 목록 조회 에러(에러 1)와는 독립적이다.

## 5. 수정 방안

### 5-1. nativeQuery에 `is_deleted` 필터 추가

```java
@Query(value = "SELECT * FROM mission m WHERE " +
    "m.is_deleted = false " +                                    // ← 추가
    "AND (:keyword IS NULL OR LOWER(CAST(m.title AS TEXT)) LIKE ...) " +
    "AND (:source IS NULL OR m.source = CAST(:source AS TEXT)) " +
    // ...
    "ORDER BY m.created_at DESC",
    countQuery = "SELECT COUNT(*) FROM mission m WHERE " +
    "m.is_deleted = false " +                                    // ← 추가
    "AND (:keyword IS NULL OR LOWER(CAST(m.title AS TEXT)) LIKE ...) " +
    // ...
    nativeQuery = true)
Page<Mission> searchMissionsAdmin(..., Pageable pageable);
```

### 5-2. Sort 충돌 해결 — Pageable에서 Sort 제거

nativeQuery에 이미 `ORDER BY m.created_at DESC`가 하드코딩되어 있으므로, Controller에서 Sort 없는 Pageable을 전달:

```java
// MissionAdminInternalController.java — 방안 A: Sort 제거
return ApiResult.<MissionAdminPageResponse>builder()
    .value(missionAdminService.searchMissions(
        keyword, source, status, type, participationType,
        creatorId, categoryId, PageRequest.of(page, size)))  // Sort 제거
    .build();
```

또는 nativeQuery의 `ORDER BY`를 제거하고 Pageable Sort를 사용하되, DB 컬럼명으로 전달:

```java
// 방안 B: nativeQuery ORDER BY 제거 + Sort에 DB 컬럼명 사용
Sort sort = "ASC".equalsIgnoreCase(sortDirection)
    ? Sort.by("created_at").ascending()    // DB 컬럼명 사용
    : Sort.by("created_at").descending();
```

**방안 A 권장**: 현재 Admin에서 정렬 옵션을 변경하는 UI가 없으므로, 하드코딩 ORDER BY 유지 + Pageable Sort 제거가 가장 안전

### 5-3. `findAllByOrderByCreatedAtDesc`에도 isDeleted 필터 추가

검색 조건이 없을 때 사용되는 JPA 메서드도 삭제된 미션을 필터링해야 함:

```java
// MissionRepository.java
Page<Mission> findAllByIsDeletedFalseOrderByCreatedAtDesc(Pageable pageable);
```
